Validation: after a successful compile, run the multi-tenant validator from your workspace, **passing the parquet path and the validation tenants explicitly**:

    uv run python {{MEWS_ROOT}}/infra/validate_all.py \
        --workspace . \
        --parquet '{{PARQUET_PATH}}' \

It invokes `./engine` for each validation tenant in turn, diffs each against DuckDB, and prints `OVERALL: PASS (all N tenants match DuckDB reference)` on success (where N = number of validation tenants). Treat its output as the truth. The single-tenant `validate.py` is still available for narrower spot-checks but is **not** the gate — only `validate_all.py` with the right `--parquet` and `--tenants` satisfies it.

If you run `validate_all.py` without `--parquet` and `--tenants`, it falls back to a synthetic Gate-1 fixture that is unrelated to your real data, will not exist on this run's parquet, and will report empty/wrong results. **Always pass both flags.**

The Mews runner runs `validate_all.py` (with the right `--parquet` and `--tenants` for this run) after this stage; if it doesn't return `OVERALL: PASS`, the stage fails regardless of what you printed.

(If your workspace sandbox restricts writes to the default uv cache dir, set `UV_CACHE_DIR=/tmp/uv_cache` on every validator invocation.)

**If validation fails or behaves unexpectedly, the issue is in your engine — not in the validator, the test environment, or system memory.** Specifically:
- **Do NOT modify** `infra/validate.py`, `infra/validate_all.py`, or any file outside your workspace. The workspace sandbox will refuse, and the refusal does not mean "try harder to bypass it" — it means "you are reaching for the wrong fix."
- **Do NOT run `sudo`** for any reason (drop_caches, install packages, change permissions, etc.). The shell sandbox will refuse. There is no scenario in which Stage 2 needs sudo.
- **Do NOT assume an OOM unless you see explicit evidence** — `Killed` in stderr, an `MemoryError` traceback, or a non-zero exit with an OOM message. `validate_all.py` peaks at ~100 MB RSS on this workload across all validation tenants combined; it does not OOM. If your engine crashes during validation, the engine is wrong, not the validator.
- **Do NOT edit `validate.py` to set DuckDB `memory_limit` or `threads`** to "make the validator happy." The validator is the truth; tuning it is moving the goalposts.
- **If a tool call comes back `TOOL_ERROR (recoverable, no state change)`**: the sandbox refused something. Read the message. Choose a different approach — re-reading the file, narrowing the failing tenant, or adjusting your engine. Do not retry the same call.

When you see a validation failure, the productive loop is: re-read `engine.cpp`, identify which tenant fails, run `./engine 'TENANT'` directly to inspect its output, compare against DuckDB's reference output for that tenant, and patch your engine.
