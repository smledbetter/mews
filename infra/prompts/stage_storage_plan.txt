Stage 1 of 3: storage plan.

Produce a storage plan that explains how you intend to lay out the spans in memory to make the locked reference query fast. This stage produces NO C++ code — only a markdown file describing your design.

Specifically, for the locked reference query, plan for:
1. The tenant filter is the outermost predicate. Choose a layout that lets the engine skip non-target tenants without scanning their rows.
2. The query joins child rows (`s.span_kind='LLM'`) to parent rows (`p.span_kind='AGENT'`) on `parent_id = span_id`. Predicate-on-both-sides plus self-join means the natural plan is two scans of different subsets of the same table; describe how you will avoid scanning the full table twice (e.g., a materialized index keyed on parent_id that already filters to AGENT-kind parents, or a denormalized `parent_kind` column written at load time).
3. `attributes.llm.model_name` is in JSON. Decode it to a flat in-memory column at load time so the per-row hot path doesn't pay JSON parsing cost.
4. The aggregation is `count(*)`, `quantile_cont(latency_ms, 0.5)`, `quantile_cont(latency_ms, 0.95)` grouped by model_name. p50/p95 typically prefer t-digest or a sorted-array sketch over full sort+pick — but for SF1 (~22k spans, ~6k LLM, ~700-1500 LLM-under-AGENT) a per-group sorted vector is small enough.

Deliverable for this stage: write `storage_plan.txt` to your workspace via `apply_patch`. Aim for roughly 80-200 lines. Cover:
- Per-column layout choices (offsets, types, packing).
- Pre-computed structures (parent_kind denormalization, model_name flattening, tenant-id partitioning).
- Concrete data structures with sizes (e.g., "uint8 span_kind enum, 8 bits, ~22 KB total at SF1").
- The execution path your stage-2 implementation will take, step by step.

Then yield. Do not implement anything. The next stage will write the C++.
